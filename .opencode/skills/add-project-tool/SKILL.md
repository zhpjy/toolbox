# Add Project Tool

当用户要求“新增工具”“做个工具”“添加一个内置工具”时，使用这个 skill。

## 项目中的工具组织方式

- 这是一个本地优先的个人工具台。
- 工具逻辑主要以 TypeScript 形式维护。
- 内置工具通常放在 `tools/builtin/<tool-id>/` 下。
- 一个工具通常对应一个独立目录，而不是散落在多个地方。
- 工具注册结果通常是生成出来的，不是只靠手工维护。

## 相关路径

- `tools/builtin/<tool-id>/`：新工具本体
- `scripts/generate-tool-registry.ts`：工具注册生成脚本
- `src/generated/tool-registry.ts`：生成后的工具注册表
- `src/generated/tool-search-index.ts`：生成后的工具搜索索引
- `package.json`：查看生成、测试、构建命令

## 内置工具内容要求

通常一个工具目录里至少会有这些文件：

- `manifest.ts`：工具元信息
- `index.ts`：工具主逻辑
- `examples.ts`：示例输入输出
- `test.ts`：最小测试
- `README.md`：简要说明

## 推荐工作流

接到“新增工具”需求后，默认按下面顺序做：

1. 先找一个最简单、最接近需求的现有工具当模板。
2. 在 `tools/builtin/<tool-id>/` 下创建工具目录和基础文件。
3. 先完成最小主逻辑，再补示例、说明和测试。
4. 执行工具生成脚本，刷新注册结果。
5. 运行该工具的最小范围测试，必要时再扩大验证。

## 实现时应知道的事实

- 这个项目对新增工具已经有比较明确的目录约定，通常不需要为了“工具应该放哪”做大量探索。
- 生成文件一般来自脚本，不应把它们当成主要手写入口。
- 如果只是新增普通文本/转换工具，通常不需要改路由层或全局 UI。
- 优先做“增量添加”，不要轻易重构整个工具体系。

## 可直接参考的项目命令

- 生成工具注册：`pnpm generate:tools`
- 测试：`pnpm test`
- 构建：`pnpm build`

如果只想做最小验证，优先运行与新增工具直接相关的测试。

## 实现时的偏好

- 优先复用现有最简单工具作为模板。
- 先做最小可用版本，再考虑补充示例、说明和测试。
- 保持实现风格贴近仓库现状，不额外引入复杂抽象。
- 如果需求文案和实际输出类型不一致，先按用户给的示例行为实现，再在结果里说明。
- 如果能通过复制一个简单工具目录再改名改内容完成，就不要先设计复杂脚手架。

## 默认检查清单

新增工具时，优先检查是否包含这些内容：

- 工具元信息
- 工具主逻辑
- 示例输入输出
- 基本测试
- 简短说明文档
- 生成后的注册结果是否已刷新

## 完成后

通常至少做这些验证：

- 执行工具注册/生成脚本
- 运行与该工具相关的测试，或最小范围测试

如果生成了自动文件，可以一并提交；如果项目约定这些文件不手改，遵循项目现状。

## 沟通要求

- 先用一句话说明准备如何落地。
- 完成后给出：改了哪些文件、做了什么验证、是否有后续可选优化。
